# EU Legal Bilingual Viewer

GDPR (English ↔ Nederlands/AVG) and the EU AI Act (English ↔ Nederlands) side by side, with hover-sync between matched paragraphs.

## What's in the repo

- `index.html` — the viewer: a single self-contained HTML+CSS+JS file, no frameworks, no build step
- `data.js` — the full bilingual content, generated by the parser
- `parse_eurlex.py` — parses official EUR-Lex HTML for both regulations in both languages and rebuilds `data.js`

## Quick start

Open `index.html` in any modern browser — everything is local, no server needed. Hover any paragraph in either column and the matching paragraph in the other language highlights. Switch between **GDPR/AVG** and **AI Act** with the top-right toggle.

## How the sync works

EU regulations share identical structure across all 24 language versions: same chapter numbers, same article numbers, same numbered paragraphs (`1.`, `2.`, …), same lettered sub-points (`(a)`, `(b)`, …). The parser builds deterministic IDs like `art-6-1-a` (Article 6, paragraph 1, sub-point a) and assigns them to both languages; the viewer's hover handler highlights every element carrying the same `data-sync` attribute.

Pairing is at paragraph/sub-point level, not sentence level — Dutch sometimes splits or merges sentences relative to English, so sentence alignment is unreliable while structural alignment is exact.

## Parsing notes (the interesting part)

Getting from EUR-Lex HTML to clean aligned data required handling a series of real-world quirks:

- **Bot protection:** EUR-Lex sits behind AWS WAF (since ~2025), which blocks scripted downloads. The parser detects the challenge page and falls back to locally saved HTML — save the four pages from your browser next to the script (`gdpr_en.html`, `gdpr_nl.html`, `aiact_en.html`, `aiact_nl.html`) and re-run.
- **Language-specific numbering styles:** Dutch sub-points use `a)` where English uses `(a)`; both regex families are handled and normalised to one ID scheme.
- **Structural edge cases:** annex boundaries (so annex text doesn't bleed into the last article), inline definition lists that needed splitting, and sub-letter ID collisions across nested lists.
- **Source-fidelity policy:** the parser reproduces EUR-Lex verbatim and logs warnings instead of "fixing" the source. Two anomalies in the published EUR-Lex text were confirmed as source errors, not parser bugs, and are preserved as published.

Run it yourself:

```bash
pip install requests beautifulsoup4 lxml
python parse_eurlex.py
```

The parser logs per-document statistics (chapters/articles/paragraphs, recital pairing, one-sided paragraphs) so any regression is visible immediately.

## Customising

Styling lives in the `<style>` block of `index.html`; CSS variables in `:root` control the palette (`--bg`, `--accent`, `--highlight`, `--highlight-strong`). The layout is responsive: under 900px the two language columns stack.

## Why not just use EUR-Lex's multilingual display?

EUR-Lex can show two languages, but without paragraph-level cross-highlighting and without pinning exactly two languages in one synchronized view. This tool is built for one specific workflow: comparing the English and Dutch wording of the *same paragraph* while doing privacy / AI-compliance work — or, if you're learning Dutch, reading formal Dutch with a perfect English crib one column away.
